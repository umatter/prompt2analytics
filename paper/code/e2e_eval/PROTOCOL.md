# End-to-End Evaluation Protocol

## 1. Overview

### Motivation

The prior evaluation (Section 5 of the paper) used 87 curated single-turn API tests to measure tool selection accuracy across LLM providers. While these tests validated that the MCP server's tool routing works correctly, peer review flagged the lack of end-to-end evidence through the actual chat interface. Specifically, the API evaluation:

- Tests only tool *selection*, not parameter extraction, numerical correctness, or interpretation quality.
- Bypasses the full application stack (Dioxus frontend, SSE streaming, dataset management).
- Uses perfectly structured JSON requests rather than natural language at varying clarity levels.
- Does not assess multi-turn conversation capabilities (follow-up questions, dataset references across turns).

### Goal

This protocol defines a systematic end-to-end evaluation where agents drive the live prompt2analytics Dioxus application through Chrome, uploading real datasets, typing prompts at varying clarity levels, and scoring full outputs against known ground truth. The evaluation measures the complete pipeline: natural language understanding, tool selection, parameter extraction, computational correctness, and interpretation quality.

## 3. Datasets

Six synthetic datasets with known data-generating processes (DGPs). All generated by `generate_datasets.R` with `set.seed(42)`.

### 3.1 eval_cross_section.csv (n=500)

**Purpose**: Cross-sectional regression with interaction effects.

**DGP**:
```
education  ~ Uniform{8, 9, ..., 22}
experience ~ Uniform{0, 1, ..., 40}
department ~ Uniform{"Engineering", "Marketing", "Finance", "Operations", "HR"}
gender     ~ Bernoulli(0.5)
salary     = 25000 + 3000*education + 1500*experience + 500*education*experience + N(0, 8000)
```

**True parameters**: intercept=25000, beta_education=3000, beta_experience=1500, beta_interaction=500, sigma=8000.

**Columns**: salary, education, experience, department, gender.

### 3.2 eval_panel.csv (N=50 firms, T=10 years = 500 rows)

**Purpose**: Panel data with firm and year fixed effects.

**DGP**:
```
value      ~ N(1000, 300)
capital    ~ N(500, 150)
sector     ~ Uniform{"Manufacturing", "Tech", "Finance", "Retail"} (firm-level)
firm_fe    ~ N(0, 100)  (50 firm effects)
year_fe    ~ N(0, 30)   (10 year effects, 2010-2019)
investment = 100 + 0.1*value + 0.05*capital + firm_fe + year_fe + N(0, 50)
```

**True parameters**: intercept=100, beta_value=0.1, beta_capital=0.05, sigma_firm=100, sigma_year=30, sigma_error=50.

**Columns**: firm_id, year, value, capital, sector, investment.

### 3.3 eval_timeseries.csv (T=200 months)

**Purpose**: Multivariate time series with AR structure, trend, and seasonality.

**DGP**:
```
gdp_growth:   AR(2) with phi1=0.6, phi2=-0.2, innovation N(0,1)
              + linear trend (0.01*t) + seasonal (0.5*sin(2*pi*t/12))
unemployment: AR(1) with phi=0.8, innovation correlated with gdp (rho=-0.5)
inflation:    AR(1) with phi=0.5, innovation correlated with unemployment (rho=0.3)
```

**True parameters**: gdp AR coefficients (0.6, -0.2), trend slope 0.01, seasonal amplitude 0.5, unemployment AR(1) phi=0.8, inflation AR(1) phi=0.5.

**Columns**: date (2005-01 to 2021-08, monthly), gdp_growth, unemployment, inflation.

### 3.4 eval_treatment.csv (n=1000 units x 2 periods = 2000 rows)

**Purpose**: Treatment effect estimation with confounding, suitable for DiD, IPW, and matching.

**DGP**:
```
x1, x2, x3 ~ N(0, 1)  (x3 is noise / potential instrument)
propensity  = logit^{-1}(0.5*x1 - 0.3*x2)
treatment   ~ Bernoulli(propensity)
outcome     = 5 + 3*treatment*period + 2*x1 + 1*x2 + N(0, 2)
```

**True treatment effect (ATT)**: 3.0 (in period 1 only; zero in period 0).

**Columns**: unit_id, period (0/1), treatment (0/1), x1, x2, x3, outcome.

### 3.5 eval_messy.csv (n=300)

**Purpose**: Data cleaning challenges -- coded column names, mixed missing indicators, inconsistent date formats.

**DGP**: Same salary model as cross_section but with obfuscation:
```
var_01 = salary (with ~5% replaced by -99 as sentinel missing)
var_02 = education as character (with ~5% replaced by "N/A")
var_03 = experience (numeric, clean)
var_04 = department (string)
var_05 = gender (0/1)
var_06 = date (mixed ISO "2023-01-15" and US "01/15/2023" formats)
```

**Key challenges**: Sentinel values (-99), string "N/A" in numeric column, mixed date formats, uninformative column names.

### 3.6 eval_survey.csv (n=400)

**Purpose**: Discrete choice and count models -- ordinal, binary, and Poisson outcomes.

**DGP**:
```
age    ~ Uniform{18, 19, ..., 75}
income ~ LogNormal(10.5, 0.7)
region ~ Uniform{"North", "South", "East", "West"}

satisfaction (ordinal 1-5):
  latent = 0.5 + 0.02*age + 0.00003*income + N(0,1)
  thresholds = (-1, 0, 1, 2)

purchased (binary):
  P(purchased=1) = logit^{-1}(-1 + 0.01*age + 0.00002*income)

visits (count):
  lambda = exp(0.5 + 0.01*age + 0.000005*income)
  visits ~ Poisson(lambda)
```

**True parameters**: satisfaction (beta_age=0.02, beta_income=0.00003), purchased (beta_age=0.01, beta_income=0.00002), visits (beta_age=0.01, beta_income=0.000005).

**Columns**: age, income, region, satisfaction (1-5), purchased (0/1), visits (count).

## 4. Test Design

### 4.1 Structure

- **32 core single-turn tests** across 8 method categories
- **3 clarity levels** per test = **96 single-turn prompts**
- **8 multi-turn conversations** (3-5 turns each) = **~30 additional turns**
- **Total**: 96 single-turn + 8 multi-turn conversations

### 4.2 Clarity Levels

Each single-turn test is written at three clarity levels:

| Level | Definition | Example (OLS regression) |
|-------|-----------|--------------------------|
| **Precise** | Names the exact method, specifies all parameters | "Run an OLS regression of salary on education and experience with HC1 robust standard errors" |
| **Moderate** | Describes the goal clearly but does not name the exact method | "Regress salary on education and experience with heteroskedasticity-robust errors" |
| **Vague** | Describes the goal loosely, omits technical details | "How do education and experience affect salary?" |

### 4.3 Method Categories

| Category | # Tests | Dataset(s) |
|----------|---------|------------|
| Regression | 5 | cross_section |
| Panel | 4 | panel |
| Time Series | 4 | timeseries |
| Causal Inference | 5 | treatment |
| Discrete Choice | 4 | survey |
| Hypothesis Testing | 4 | cross_section, panel |
| Data Munging | 3 | messy |
| Visualization | 3 | cross_section, timeseries |
| **Total** | **32** | |

## 6. Multi-Turn Sequences

Eight multi-turn conversations testing the system's ability to maintain context across turns.

### MT1: Exploratory to Regression (3 turns)
- **Dataset**: eval_cross_section.csv
- T1: "Load eval_cross_section.csv and show me a summary of the data."
- T2: "What's the relationship between education and salary? Show me a scatter plot."
- T3: "Now run a regression of salary on education and experience with robust standard errors."

### MT2: Panel Data Workflow (4 turns)
- **Dataset**: eval_panel.csv
- T1: "Load eval_panel.csv and describe the panel structure."
- T2: "Estimate a fixed effects model of investment on value and capital."
- T3: "Now estimate random effects and compare with a Hausman test."
- T4: "Re-estimate the fixed effects model with firm-clustered standard errors."

### MT3: Time Series Pipeline (4 turns)
- **Dataset**: eval_timeseries.csv
- T1: "Load eval_timeseries.csv and plot the gdp_growth series."
- T2: "Check the ACF and PACF of gdp_growth."
- T3: "Based on those results, fit an appropriate ARIMA model."
- T4: "Now run a VAR with gdp_growth and unemployment. Test for Granger causality."

### MT4: Causal Inference Chain (5 turns)
- **Dataset**: eval_treatment.csv
- T1: "Load eval_treatment.csv. How many treated vs control units are there?"
- T2: "Check for covariate balance between treatment and control groups."
- T3: "Estimate the treatment effect using difference-in-differences."
- T4: "Now estimate using propensity score matching. How does the ATT compare?"
- T5: "Finally, try IPW estimation. Summarize all three estimates."

### MT5: Data Cleaning Workflow (4 turns)
- **Dataset**: eval_messy.csv
- T1: "Load eval_messy.csv and describe the data. What problems do you see?"
- T2: "var_01 is salary and var_02 is education. Replace -99 in salary and N/A in education with missing values."
- T3: "Now rename all columns to meaningful names: var_03=experience, var_04=department, var_05=gender, var_06=hire_date."
- T4: "Run a regression of salary on education and experience."

### MT6: Discrete Choice Sequence (4 turns)
- **Dataset**: eval_survey.csv
- T1: "Load eval_survey.csv and show the distribution of the satisfaction variable."
- T2: "Fit an ordered logit model predicting satisfaction from age and income."
- T3: "Now fit a logistic regression predicting purchased from age and income."
- T4: "Finally, fit a Poisson model for visits using age and income as predictors."

### MT7: Diagnostic-Driven Respecification (3 turns)
- **Dataset**: eval_cross_section.csv
- T1: "Regress salary on education and experience."
- T2: "Run diagnostic tests on those results -- check for heteroskedasticity and normality of residuals."
- T3: "Based on the diagnostics, re-estimate with appropriate robust standard errors."

### MT8: Cross-Dataset Comparison (3 turns)
- **Datasets**: eval_cross_section.csv, eval_panel.csv
- T1: "Load eval_cross_section.csv and run OLS of salary on education and experience."
- T2: "Now load eval_panel.csv and run a fixed effects model of investment on value and capital."
- T3: "Compare the R-squared values from both models. Which has better explanatory power?"

## 7. Scoring Rubric

Each single-turn prompt is scored on four dimensions, each rated 0-2:

### 7.1 Tool Selection (0-2)

| Score | Criteria |
|-------|----------|
| 2 | Selects the exact correct tool (e.g., `ols_regression` for OLS) |
| 1 | Selects an acceptable alternative tool (e.g., `panel_fixed_effects` when `ols_regression` with dummy variables would also work) |
| 0 | Selects a wrong or irrelevant tool, or fails to call any tool |

### 7.2 Parameter Extraction (0-2)

| Score | Criteria |
|-------|----------|
| 2 | All parameters correctly extracted: dataset name, dependent variable, independent variables, options (robust SEs, clustering, etc.) |
| 1 | Core parameters correct (dataset, y, x) but optional parameters missing or wrong (e.g., forgot to specify robust SEs when asked) |
| 0 | Core parameters wrong (wrong dependent variable, wrong dataset, missing key variables) |

### 7.3 Numerical Correctness (0-2 or N/A)

| Score | Criteria |
|-------|----------|
| 2 | Reported values within tolerance of true DGP parameters or reference R output |
| 1 | Reasonable values (correct sign, right order of magnitude) but outside tolerance |
| 0 | Wrong values (wrong sign, wrong magnitude, or computation error) |
| N/A | Not applicable (visualization, data loading, data description tasks) |

### 7.4 Interpretation Quality (0-2)

| Score | Criteria |
|-------|----------|
| 2 | Accurate interpretation with relevant context (e.g., "education coefficient of ~3000 means each additional year of education is associated with ~$3000 higher salary, holding experience constant") |
| 1 | Superficial but not wrong interpretation (e.g., "the coefficient is statistically significant") |
| 0 | Misleading or incorrect interpretation (e.g., claiming causal effect from OLS, wrong direction of effect) |

### 7.5 Scoring Summary

- **Maximum score per prompt**: 8 points (4 dimensions x 2 points)
- **Adequate threshold**: >= 5 points
- **For N/A dimensions**: Score is normalized to (earned / max_applicable) x 8

### 7.6 Multi-Turn Scoring

Multi-turn conversations are scored per-turn using the same rubric, plus:
- **Context retention**: Does the model correctly reference prior results/datasets? (binary: yes/no per turn)
- **Conversation completion**: Did the model successfully complete all turns? (binary: yes/no)

## 8. Execution Protocol

### 8.1 Prerequisites

1. **Build and start the backend**:
   ```bash
   cargo build --release -p p2a-mcp
   cargo run -p p2a-mcp --features full -- --transport http --host 127.0.0.1 --port 8080 --cors-permissive
   ```

2. **Start the Dioxus frontend**:
   ```bash
   cd crates/p2a-dioxus && dx serve
   ```

3. **For local models, start Ollama**:
   ```bash
   ollama serve
   ollama pull qwen2.5:7b
   ollama pull ministral:3b
   ```

4. **Open Chrome** and navigate to the Dioxus app (typically `http://localhost:8080` or the dx serve URL).

5. **Install/enable Claude-in-Chrome extension** for automated interaction.

### 8.2 Per-Model Procedure

For each model:

1. Open the Dioxus app Settings panel.
2. Select the LLM provider and model.
3. Verify connectivity (send a test message).
4. Clear all loaded datasets and conversation history.

### 8.3 Per-Test Procedure (Single-Turn)

For each test prompt:

1. **Start a new conversation** (click "New Chat" or equivalent).
2. **Upload the required dataset** if not already loaded (drag-and-drop or file picker).
3. **Type the prompt** exactly as specified in `test_cases.json`.
4. **Wait for the complete response** (all tool calls finished, streaming complete).
5. **Record the result** in the result JSON (see Section 8.5).
6. **Do NOT clear the dataset** between tests using the same dataset -- only clear when switching datasets.

### 8.4 Per-Test Procedure (Multi-Turn)

For each multi-turn conversation:

1. **Start a new conversation**.
2. **Upload the required dataset(s)**.
3. **Send each turn's prompt** in sequence, waiting for the complete response before proceeding.
4. **Record each turn separately** in the result JSON.
5. **Note context retention** issues (model forgetting previous dataset, results, or variable names).

### 8.5 Between-Model Procedure

Between models:
1. Clear all datasets from the backend.
2. Clear all conversation history.
3. Switch the model in Settings.
4. Verify the new model responds correctly to a simple test prompt.

### 8.6 Result Recording Format

Each test result is recorded as a JSON object:

```json
{
  "test_id": "ST01_precise",
  "model": "claude-sonnet-4-6",
  "timestamp": "2026-03-07T14:30:00Z",
  "prompt": "Run an OLS regression of salary on education and experience with HC1 robust standard errors using the eval_cross_section dataset.",
  "tool_calls": [
    {
      "tool": "ols_regression",
      "parameters": {
        "dataset": "eval_cross_section",
        "y": "salary",
        "x": ["education", "experience"],
        "robust": "hc1"
      }
    }
  ],
  "response_text": "...",
  "scores": {
    "tool_selection": 2,
    "parameter_extraction": 2,
    "numerical_correctness": 2,
    "interpretation": 2
  },
  "total_score": 8,
  "adequate": true,
  "notes": ""
}
```

Multi-turn results include an additional `turns` array:

```json
{
  "test_id": "MT1",
  "model": "claude-sonnet-4-6",
  "turns": [
    {
      "turn": 1,
      "prompt": "...",
      "tool_calls": [...],
      "scores": {...},
      "context_retained": true
    },
    ...
  ],
  "conversation_completed": true
}
```

## 9. Metrics

### 9.1 Primary Metrics

| Metric | Definition |
|--------|-----------|
| **Overall adequate rate** | % of single-turn tests scoring >= 5/8 |
| **By clarity level** | Adequate rate for precise / moderate / vague prompts |
| **By method category** | Adequate rate per category (regression, panel, etc.) |

### 9.2 Dimension-Level Metrics

| Metric | Definition |
|--------|-----------|
| **Tool selection accuracy** | % of tests scoring 2 on tool_selection |
| **Parameter extraction rate** | % of tests scoring 2 on parameter_extraction |
| **Numerical accuracy** | % of applicable tests scoring 2 on numerical_correctness |
| **Interpretation quality** | Mean interpretation score (0-2 scale) |

### 9.3 Multi-Turn Metrics

| Metric | Definition |
|--------|-----------|
| **Multi-turn completion rate** | % of conversations where all turns completed successfully |
| **Context retention rate** | % of follow-up turns where prior context was correctly used |
| **Per-turn adequate rate** | % of individual turns scoring >= 5/8 |

### 9.4 Special Metrics

| Metric | Definition |
|--------|-----------|
| **Messy data handling** | Adequate rate on ST27-ST29 (data cleaning tests) |
| **Vague prompt recovery** | Adequate rate on vague-level prompts specifically |

### 9.5 Reporting

Results are aggregated into tables for the paper:
- Model x clarity level matrix (adequate rates)
- Model x category matrix (adequate rates)
- Model x dimension breakdown (mean scores)
- Multi-turn completion summary

### 10.4 Reproducibility Notes

- All synthetic datasets generated with `set.seed(42)` in R.
- The `generate_datasets.R` script is deterministic and produces identical CSV files on any platform.
- Test prompts are fixed in `test_cases.json` -- no variation across runs.
- LLM responses are inherently non-deterministic; we record the temperature setting used (default for each provider) but do not expect exact reproducibility of LLM outputs.
- Scoring involves human judgment; inter-rater reliability should be assessed if multiple scorers are used.
