\section{Examples} \label{sec:examples}

This section demonstrates \pkg{prompt2analytics} through
two interfaces: the natural language chat interface powered by large language
models, and the command-line interface for scripted workflows. Both interfaces
access the same underlying \proglang{Rust} analytics engine.

We use two datasets throughout this section:
\begin{itemize}
  \item The Grunfeld investment data \citep{Baltagi:2013}, a classic panel
    dataset with 200 observations (10 firms over 20 years) containing firm
    investment, market value, and capital stock. This dataset is included
    in the \pkg{plm} package and can be loaded via \code{data("Grunfeld",
    package = "plm")} in \proglang{R}; we provide it as \code{grunfeld.csv}
    in the supplementary materials.
  \item A simulated cross-sectional dataset demonstrating basic regression
    analysis with heteroskedasticity.
\end{itemize}


%% -----------------------------------------------------------------------------
\subsection{Chat interface} \label{sec:examples-chat}

The chat interface enables natural language interaction with the analytics
engine. Users describe their analysis goals in plain language, and the LLM
translates these requests into structured tool calls executed by the MCP server.
This interface is useful for exploratory data analysis and for
users less familiar with programming syntax. The examples in this section
were generated using Claude 3.5 Sonnet via the MCP integration; other models
(GPT-4o, Llama 3.1 8B, etc.) produce similar results with minor variations
in explanation style.

\subsubsection{Loading and exploring data}

A typical session begins by loading data and examining its structure:

\begin{Sinput}
User: Load the Grunfeld investment dataset from grunfeld.csv and show me
      what's in it.
\end{Sinput}

\begin{Soutput}
I'll load the dataset and describe its contents.

**Dataset loaded: grunfeld**
- Rows: 200
- Columns: 5

| Column  | Type    | Description                        |
|---------|---------|-----------------------------------|
| firm    | Integer | Firm identifier (1-10)            |
| year    | Integer | Year (1935-1954)                  |
| inv     | Float64 | Gross investment                  |
| value   | Float64 | Market value of firm              |
| capital | Float64 | Stock of plant and equipment      |

This is a balanced panel dataset with 10 firms observed over 20 years,
commonly used to demonstrate panel data methods.
\end{Soutput}

\subsubsection{OLS regression with robust standard errors}

Cross-sectional regression with heteroskedasticity-consistent standard errors.%
\footnote{HC1 is the default robust standard error estimator when robust SEs are
requested. HC1 applies a degrees-of-freedom correction $(n/(n-k))$ to HC0,
improving small-sample performance.}

\begin{Sinput}
User: Run a regression of investment on market value and capital stock.
      Use robust standard errors.
\end{Sinput}

\begin{Soutput}
Running OLS regression with HC1 robust standard errors...

**OLS Regression Results**
Dependent variable: inv

| Variable    |    Coef |  Std Err |      t | P>|t| |
|-------------|--------:|---------:|-------:|------:|
| (Intercept) | -42.714 |   11.575 | -3.690 | 0.000 |
| value       |   0.116 |    0.007 | 16.967 | 0.000 |
| capital     |   0.231 |    0.049 |  4.721 | 0.000 |

R-squared: 0.812
Adj. R-squared: 0.810
Observations: 200
Standard errors: HC1 (heteroskedasticity-robust)

Both market value and capital stock are highly significant predictors of
investment. A $1 million increase in market value is associated with
approximately $116,000 more investment, holding capital constant.
\end{Soutput}

\subsubsection{Panel data: Fixed effects estimation}

For panel data, users can request fixed effects estimation in natural language:

\begin{Sinput}
User: The data has a panel structure. Run a fixed effects regression
      controlling for firm-specific effects.
\end{Sinput}

\begin{Soutput}
Running within-transformation fixed effects estimation...

**Fixed Effects (Within) Estimation**
Dependent variable: inv
Entity fixed effects: firm (10 levels)

| Variable |    Coef |  Std Err |      t | P>|t| |
|----------|--------:|---------:|-------:|------:|
| value    |   0.110 |    0.012 |  9.288 | 0.000 |
| capital  |   0.310 |    0.017 | 17.867 | 0.000 |

R-squared (within): 0.767
Observations: 200
Groups: 10
F-statistic: 309.01 (p-value: 0.000)

After controlling for time-invariant firm characteristics, the coefficient
on capital increases from 0.23 to 0.31, suggesting that pooled OLS
underestimated the association between capital stock and investment.
(Note: This change in coefficient reflects removal of omitted variable
bias, not necessarily identification of a causal effect.)
\end{Soutput}

\subsubsection{Model comparison: Hausman test}

Users can request specification tests without knowing the technical details:

\begin{Sinput}
User: Should I use fixed effects or random effects for this analysis?
\end{Sinput}

\begin{Soutput}
I'll run a Hausman test to help determine the appropriate specification.

**Hausman Test Results**
H0: Random Effects is consistent and efficient
H1: Fixed Effects is consistent; Random Effects is inconsistent

Chi-squared statistic: 0.0003
Degrees of freedom: 2
P-value: 0.9999

**Recommendation**: Random Effects

The test fails to reject the null hypothesis. The near-zero chi-squared
statistic indicates virtually no systematic difference between the FE and
RE coefficient estimates for this dataset.

**Note**: Extreme p-values like 0.9999 warrant interpretation with caution.
Such values can arise from small test statistics (as here), but also from
numerical issues or model misspecification. The Hausman test \citep{Hausman:1978}
assumes both estimators are consistent under the null, which requires correct
functional form, homoskedastic errors, and no omitted variables correlated
with the regressors. The test statistic may be unreliable with small samples,
heteroskedasticity, or serial correlation. Users should treat very high
p-values as suggestive rather than definitive.
\end{Soutput}

| capital  |   0.358 |    0.054 |  6.579 | 0.000 |

R-squared (within): 0.720
Observations: 200
F-statistic: 217.44 (p-value: 0.000)
\end{Soutput}

\subsection{Command-line interface} \label{sec:examples-cli}

The command-line interface provides programmatic access for scripted workflows.
Commands follow a hierarchical structure with sessions maintaining state:

\begin{Sinput}
# Load data and run OLS with robust standard errors
p2a --session analysis.json data load grunfeld.csv --name grunfeld
p2a --session analysis.json reg ols grunfeld \
    -y inv -x value capital --robust hc1

# Panel fixed effects
p2a --session analysis.json panel fe grunfeld \
    -y inv -x value capital --entity firm

# Export session to reproducible script
p2a script export analysis.json -o reproduce.sh
\end{Sinput}
\noindent\textit{Note}: Backslash line continuation (\texttt{\textbackslash})
works in Bash, Zsh, and most POSIX-compatible shells. Windows Command Prompt
users should write commands on a single line or use PowerShell's backtick
(\texttt{`}) for continuation.

All commands support JSON output (\code{-F json}) for programmatic integration.
Appendix~\ref{app:extended-examples} provides additional CLI examples including
panel estimation, hypothesis tests, and two-way fixed effects.

\subsection{Comparison with R} \label{sec:examples-comparison}

For reference, the equivalent \proglang{R} code for the fixed effects
estimation is:

\begin{Sinput}
library(plm)
data("Grunfeld", package = "plm")

# Fixed effects model
fe_model <- plm(inv ~ value + capital,
                data = Grunfeld,
                index = c("firm", "year"),
                model = "within")
summary(fe_model)

# Two-way fixed effects
fe_twoway <- plm(inv ~ value + capital,
                 data = Grunfeld,
                 index = c("firm", "year"),
                 model = "within",
                 effect = "twoways")
summary(fe_twoway)
\end{Sinput}

The \pkg{prompt2analytics} implementation produces numerically equivalent
results (see Section~\ref{sec:comparison}), while offering the conversational
interface that enables users unfamiliar with \proglang{R} syntax to perform
the same analyses through natural language.

\subsection{Multi-step workflows} \label{sec:examples-multistep}

Real-world analysis involves multiple steps: data inspection, handling issues,
and specification refinement. The chat interface maintains context across turns,
enabling progressive model refinement guided by user questions. The LLM provides
interpretive commentary while all computations execute through validated
\proglang{Rust} implementations.

Appendix~\ref{app:extended-examples} provides a detailed multi-step workflow

\subsection{Realistic interaction patterns} \label{sec:examples-realistic}

The preceding examples show things working. Real use involves more back-and-forth.
With tool calls in conversation history, multi-turn accuracy is 83--97\%
(Section~\ref{sec:local}). Parameter extraction is less reliable (F1 of
41--59\%), so check what the model extracted. This section shows what failures
look like and how to recover.

\subsubsection{Ambiguous specification requiring clarification}

\begin{Sinput}
User: Run a regression with firm effects.
\end{Sinput}

\begin{Soutput}
I can estimate this model, but need clarification:

1. **Dependent variable**: Which column should be the outcome?
2. **Independent variables**: Which predictors should be included?
3. **Fixed vs Random effects**: Should firm effects be treated as fixed
   (within estimator) or random (GLS)?

Please specify, e.g., "Regress inv on value and capital with firm fixed effects"
\end{Soutput}

\begin{Sinput}
User: Regress inv on value with firm fixed effects.
\end{Sinput}

\begin{Soutput}
**Fixed Effects (Within) Estimation**
Dependent variable: inv
Entity fixed effects: firm (10 levels)

| Variable |    Coef |  Std Err |      t | P>|t| |
|----------|--------:|---------:|-------:|------:|
| value    |   0.095 |    0.010 |  9.747 | 0.000 |

R-squared (within): 0.334
\end{Soutput}

\subsubsection{Incorrect method selection}

When users describe analyses ambiguously, the LLM may select inappropriate methods:

\begin{Sinput}
User: Estimate the causal effect of capital on investment.
\end{Sinput}

\begin{Soutput}
**OLS Regression Results**
Dependent variable: inv

| Variable    |    Coef |  Std Err |      t | P>|t| |
|-------------|--------:|---------:|-------:|------:|
| (Intercept) | -17.892 |   11.312 | -1.582 | 0.115 |
| capital     |   0.234 |    0.012 | 19.524 | 0.000 |

A one-unit increase in capital is associated with a 0.234 increase in
investment.
\end{Soutput}

\begin{Sinput}
User: That's just correlation. I need the causal effect - capital is
      endogenous due to simultaneity with investment decisions.
\end{Sinput}

\begin{Soutput}
You're right - capital and investment are jointly determined, making OLS
estimates biased. For causal identification, you would need:

1. **Instrumental Variables**: A valid instrument affecting capital but
   not investment directly
2. **Panel methods**: Firm fixed effects can address time-invariant
   confounding, though not simultaneity
3. **Lagged variables**: Using lagged capital may reduce (but not eliminate)
   simultaneity bias

The system implements IV/2SLS if you have instruments. Without instruments,
I can only provide associational estimates with appropriate caveats.

**Important**: This system executes statistical methods but does not validate
identification strategies. Causal claims require justification beyond the
statistical procedure.
\end{Soutput}

\subsubsection{Parameter extraction failure}

The LLM sometimes extracts parameters incorrectly from natural language:

\begin{Sinput}
User: Use HC2 standard errors and cluster by firm.
\end{Sinput}

\begin{Soutput}
**OLS Regression Results** (HC2 robust standard errors)
...
\end{Soutput}

\begin{Sinput}
User: I asked for clustering by firm, not just robust SEs.
\end{Sinput}

\begin{Soutput}
I apologize for the error. Let me re-run with clustered standard errors:

**OLS Regression Results** (Clustered by firm, 10 clusters)

| Variable    |    Coef | Cluster SE |      t | P>|t| |
|-------------|--------:|-----------:|-------:|------:|
| (Intercept) | -42.714 |     23.624 | -1.808 | 0.104 |
| value       |   0.116 |      0.013 |  8.844 | 0.000 |
| capital     |   0.231 |      0.068 |  3.409 | 0.008 |

**Note**: With only 10 clusters, clustered standard errors may be
downward biased. Consider wild cluster bootstrap for more reliable inference.
\end{Soutput}

\subsubsection{What to expect}

From our evaluation:
\begin{itemize}
  \item Multi-turn workflows complete 60--90\% of the time when tool calls
        are in history (Section~\ref{sec:local})
  \item Naming variables directly (``regress inv on value'') beats pronouns
        (``use the same variables'')
  \item Check extracted parameters before trusting output
\end{itemize}

The fix for most failures: keep tool calls in conversation history, and glance
at what the model extracted.

\subsection{A realistic multi-turn workflow} \label{sec:examples-failure}

The preceding examples show individual exchanges. Here is a full multi-turn
workflow showing what can go wrong: wrong method selected, parameters
extracted incorrectly, user corrections, eventual success.

\begin{Sinput}
User: Analyze whether the training program affected employee performance.
      The data has employees observed before and after training, with a
      control group that didn't receive training.
\end{Sinput}

\begin{Soutput}
I'll run a t-test comparing performance between training and control groups.

**Two-sample t-test**
Training group mean: 3.82
Control group mean: 3.45
Difference: 0.37 (95% CI: [0.21, 0.53])
p-value: 0.000

The training group shows significantly higher performance.
\end{Soutput}

\begin{Sinput}
User: That's not right - I need a difference-in-differences analysis,
      not just a comparison of means. I have before/after data.
\end{Sinput}

\begin{Soutput}
You're right, I apologize. Let me run a difference-in-differences estimation
to properly account for the before/after structure.

**Difference-in-Differences Estimation**
Treatment indicator: training_group
Time indicator: post_training

| Variable              |    Coef |  Std Err |      t | P>|t| |
|-----------------------|--------:|---------:|-------:|------:|
| post_training         |   0.12  |    0.08  |  1.50  | 0.134 |
| training_group        |   0.31  |    0.11  |  2.82  | 0.005 |
| post × training (ATT) |   0.24  |    0.09  |  2.67  | 0.008 |

The ATT estimate is 0.24 performance points.
\end{Soutput}

\begin{Sinput}
User: Good, but I asked for clustered standard errors at the employee
      level to account for repeated observations.
\end{Sinput}

\begin{Soutput}
Let me re-run with clustered standard errors:

**Difference-in-Differences** (Clustered by employee, 847 clusters)

| Variable              |    Coef | Cluster SE |      t | P>|t| |
|-----------------------|--------:|-----------:|-------:|------:|
| post_training         |   0.12  |      0.11  |  1.09  | 0.276 |
| training_group        |   0.31  |      0.14  |  2.21  | 0.027 |
| post × training (ATT) |   0.24  |      0.12  |  2.00  | 0.046 |

With clustered SEs, the ATT remains significant at the 5% level, though
standard errors are larger than the unclustered estimates.
\end{Soutput}

\begin{Sinput}
User: Now check for pre-trends to validate the parallel trends assumption.
\end{Sinput}

\begin{Soutput}
**Pre-trend Test**
Testing whether treatment and control groups had parallel trends
in the pre-treatment period...

| Pre-period |  Diff |  Std Err |      t | P>|t| |
|------------|------:|---------:|-------:|------:|
| t-3        | -0.02 |    0.09  | -0.22  | 0.826 |
| t-2        |  0.05 |    0.08  |  0.63  | 0.531 |
| t-1        |  0.01 |    0.08  |  0.13  | 0.900 |

Joint test (H0: all pre-period diffs = 0):
F-statistic: 0.18, p-value: 0.912

No evidence of differential pre-trends; parallel trends assumption
is supported (though not proven) by this test.
\end{Soutput}

This workflow required three correction turns: (1)~correcting method selection
from t-test to DiD; (2)~adding clustered standard errors; and (3)~requesting
pre-trend diagnostics. The final result is correct. With tool calls in
conversation history, these corrections become less common: completion rates
are 60--90\% versus 0--35\% without (Section~\ref{sec:local}).

\subsection{Error handling and recovery} \label{sec:examples-errors}

The system handles common failure modes with explanatory guidance. Two
examples illustrate the approach.

\subsubsection{Multicollinearity detection}

\begin{Sinput}
User: Add years_since_hire to the model alongside tenure.
\end{Sinput}

\begin{Soutput}
**Error: Perfect multicollinearity detected**

The design matrix is rank-deficient. Column `years_since_hire` is a linear
combination of existing columns (correlation with tenure: 1.000).

**Suggested actions:**
1. Remove one of the collinear variables
2. If both are theoretically important, consider which better captures
   your construct of interest
\end{Soutput}

\subsubsection{Convergence failure}

\begin{Sinput}
User: Fit a logit model with department fixed effects.
\end{Sinput}

\begin{Soutput}
**Warning: Convergence issues detected**

**Issue:** Perfect prediction in some groups
- Department "Legal" has 0% bonus rate (23 observations)

**Options:**
1. Drop singleton departments
2. Use a linear probability model (robust to separation)
3. Use penalized likelihood (Firth correction)

Running linear probability model with department FE instead...
\end{Soutput}

The system detects problems before estimation fails, explains statistical
issues rather than computational symptoms, and provides actionable suggestions.
When requests exceed capabilities, it clearly states limitations and points
to external tools. Appendix~\ref{app:extended-examples} provides additional
error handling examples including small cluster warnings and out-of-scope
request handling.
