\section{Software architecture} \label{sec:architecture}

\pkg{prompt2analytics} is organized as a \proglang{Rust} workspace with
four crates that separate core computation, user interfaces,
and LLM integration. Figure~\ref{fig:architecture} illustrates the
architecture, showing two pathways: direct access via the CLI, and
LLM-mediated access through chat interfaces.

\begin{description}
  \item[\pkg{p2a-core}] Core analytics library implementing all statistical
    methods in pure \proglang{Rust}. This crate has no external econometrics
    dependencies; all estimators are implemented from first principles using
    \pkg{ndarray} for matrix operations and \pkg{faer} for numerically stable
    linear algebra. Implementations follow standard textbook algorithms:
    OLS and robust standard errors follow \cite{Wooldridge:2010} and
    \cite{Cameron+Trivedi:2005}; panel data methods follow \cite{Baltagi:2013};
    time series methods follow \cite{Hamilton:1994}; maximum likelihood
    estimation for discrete choice models uses Newton-Raphson as described
    in \cite{Greene:2018}.
  \item[\pkg{p2a-cli}] Command-line interface exposing the full analytics
    functionality through hierarchical subcommands. Produces a standalone
    \code{p2a} binary.
  \item[\pkg{p2a-mcp}] Model Context Protocol server exposing 101 tools for
    LLM integration. Supports multiple transport mechanisms (stdio, HTTP,
    WebSocket) and maintains session state for dataset persistence.
  \item[\pkg{p2a-dioxus}] Web application written entirely in \proglang{Rust}
    using the Dioxus framework, compiling to WebAssembly for browser deployment.
    Communicates with the MCP server via HTTP.
\end{description}

All user-facing crates depend on \pkg{p2a-core}, ensuring that statistical
computations are identical regardless of the interface used.

\subsubsection*{Quick start}

The simplest way to use \pkg{prompt2analytics} is through the CLI:

\begin{Sinput}
# Load data and run OLS with robust standard errors
p2a data load wages.csv --name wages
p2a reg ols wages -y salary -x education experience --robust hc1
\end{Sinput}

\noindent For LLM-mediated access, configure the MCP server and type in
natural language:

\begin{Sinput}
User: Load wages.csv and regress salary on education and experience
      with robust standard errors.
\end{Sinput}

\noindent Both pathways invoke the same \proglang{Rust} implementation and
produce identical numerical results.

\subsection{Core library organization}

The \pkg{p2a-core} library organizes modules by analytical workflow: linear
algebra primitives (\code{linalg}), regression methods with robust standard
errors (\code{regression}), panel/IV/DiD/discrete choice methods
(\code{econometrics}), 40+ statistical tests (\code{stats}), time series
and forecasting (\code{forecasting}), clustering and dimensionality reduction
(\code{ml}), and visualization (\code{visualization}).

A \code{LinearEstimator} trait provides a consistent interface across OLS,
panel models, and IV estimators:

\begin{Code}
pub trait LinearEstimator {
    fn coefficients(&self) -> &Array1<f64>;
    fn std_errors(&self) -> &Array1<f64>;
    fn t_values(&self) -> Array1<f64>;    // default impl
    fn p_values(&self) -> Array1<f64>;    // default impl
    fn residuals(&self) -> Array1<f64>;
    fn n_obs(&self) -> usize;
    fn df(&self) -> usize;
}
\end{Code}

\noindent Implementors provide core results (coefficients,
standard errors, residuals); derived statistics ($t$-values, $p$-values,
confidence intervals) are computed via default implementations. This ensures
comparable output across estimators while enforcing the interface at compile
time. For example, calling OLS from \proglang{Rust} code:

\begin{Code}
use p2a_core::regression::ols::run_ols;
use p2a_core::regression::ols::CovarianceType;

let result = run_ols(
    &dataset, "inv", &["value", "capital"],
    true,                    // intercept
    CovarianceType::HC1,     // robust SEs
)?;
println!("R-squared: {:.4}", result.r_squared);
for (name, coef, se) in result.summary() {
    println!("{}: {:.4} ({:.4})", name, coef, se);
}
\end{Code}

\noindent Unlike \proglang{R}'s formula interface, \pkg{p2a-core} uses an explicit
column-based API, since the LLM already translates intent to specification.

The \code{EconError} type provides specific, actionable error messages
(singular matrix, perfect multicollinearity, convergence failure, insufficient
clusters) that enable the LLM to provide helpful recovery guidance.

\subsection{User interfaces}

Three interfaces provide access to the core functionality, as shown in
Figure~\ref{fig:architecture}.

\subsubsection{Command-line interface}

The CLI uses hierarchical subcommands organized by analytical domain:

\begin{verbatim}
p2a data load sales.csv --name sales
p2a reg ols sales -y revenue -x advertising budget --robust hc1
p2a panel fe sales -y revenue -x advertising --entity firm --time year
p2a viz scatter sales -x advertising -y revenue -o scatter.png
\end{verbatim}

Session management enables dataset persistence across commands and script
generation for reproducibility. The \code{--session} flag records all
commands to a JSON file that can be exported as an executable script.

\subsubsection{MCP server}

The MCP server exposes 101 tools organized into categories that mirror
the core library structure:

\begin{center}
\begin{tabular}{llr}
\toprule
Category & Examples & Tools \\
\midrule
Data management & \code{load\_dataset}, \code{describe\_dataset} & 7 \\
Data transformation & \code{munge\_filter}, \code{munge\_join}, \code{munge\_pivot} & 19 \\
Regression & \code{regression\_ols}, \code{regression\_diagnostics} & 10 \\
Panel data & \code{panel\_fixed\_effects}, \code{hausman\_test} & 5 \\
Causal inference & \code{iv\_2sls}, \code{diff\_in\_diff} & 8 \\

Visualization & \code{viz\_scatter}, \code{viz\_histogram\_interactive} & 7 \\
\bottomrule
\end{tabular}
\end{center}

Each tool has a JSON Schema defining required and optional parameters,
enabling LLMs to generate valid tool calls. The schema design follows
several principles: required parameters are minimized to reduce LLM errors,
sensible defaults are provided (e.g., HC1 robust standard errors), and
parameter names use consistent conventions (dataset identifier is always
\code{dataset}, dependent variable is \code{y\_var}). Tool descriptions
are written to be interpretable by LLMs, specifying when each tool is
appropriate and what alternatives exist. For example, the OLS tool schema is:

\begin{verbatim}
{
  "name": "regression_ols",
  "description": "Run OLS regression with optional robust SEs",
  "inputSchema": {
    "required": ["dataset", "y_var", "x_vars"],
    "properties": {
      "dataset": {"type": "string"},
      "y_var": {"type": "string"},
      "x_vars": {"type": "array", "items": {"type": "string"}},
      "robust": {"type": "string", "enum": ["hc0","hc1","hc2","hc3"]},
      "cluster_var": {"type": "string"}
    }
  }
}
\end{verbatim}

\noindent The complete tool schema is available in the supplementary
materials (\code{mcp\_tool\_schema.json}).

Note the distinction between \emph{MCP tools} and \emph{core methods}: the
101 MCP tools serve as entry points that may invoke one or more of the 200+
statistical methods in \pkg{p2a-core}. For instance, the
\code{regression\_ols} tool provides access to OLS estimation with multiple
covariance estimators (HC0--HC3, clustered, HAC), each a distinct method
in the core library. This many-to-one mapping keeps the LLM's tool selection
task manageable while preserving access to the full method library.

The server supports multiple transport mechanisms: stdio for direct CLI
integration, HTTP for the web application, and WebSocket for streaming results.

\subsubsection{Web application}

The web interface is implemented entirely in \proglang{Rust} using the
Dioxus framework, which compiles to WebAssembly for browser deployment.
This pure-\proglang{Rust} approach ensures type safety across the full
stack and eliminates the JavaScript build toolchain. The application
communicates with the MCP server's HTTP endpoint and supports three
LLM providers: Ollama (local), Claude (Anthropic), and GPT (OpenAI).
Conversations are persisted via an embedded SurrealDB database on the
server side.

\subsection{Communication flow}

Figure~\ref{fig:chat-flow} illustrates the interaction pattern for
LLM-mediated requests. When a user submits a natural language query:

\begin{enumerate}
  \item The query is sent to the LLM provider (cloud or local Ollama)
  \item The LLM interprets the request and generates a structured tool call
    with appropriate parameters
  \item The MCP server routes the call to the corresponding \proglang{Rust}
    method in \pkg{p2a-core}
  \item The core library loads data and performs computation locally
  \item Results (coefficients, test statistics, diagnostics) are serialized
    to JSON and returned through the MCP server
  \item The LLM receives the results and formulates a natural language response
  \item The response is displayed to the user
\end{enumerate}

\begin{figure}[htbp]
\centering
\includegraphics[width=\textwidth]{figures/chat_flow_v2}
\caption{Sequence diagram showing the chat interaction flow. The user's
natural language request is interpreted by the LLM, which generates a

This architecture maintains data locality: the full dataset never leaves
the user's machine. Only metadata (column names, types), tool parameters,
and aggregated results pass through the LLM. Data previews do transmit
sample rows; for complete isolation, local LLM deployment via Ollama
eliminates all external data transmission.

\subsection{Technical design decisions}

Several technical choices merit brief explanation.

\textbf{Numerical precision.} All computations use 64-bit floating point
(\code{f64}) throughout, providing approximately 15--17 significant decimal
digits. This matches the precision used by \proglang{R} and most established
statistical software. Extended precision (128-bit) is not supported; users
requiring higher precision for specialized applications (certain financial
calculations, cryptographic contexts, or severely ill-conditioned matrices)
should use dedicated software.

\textbf{Conversation persistence.} The web application uses SurrealDB, an
embedded document database, for storing conversation history and settings.
SurrealDB was chosen over SQLite for two reasons: (1)~its embedded operation
requires no external service, enabling zero-configuration deployment where the
database runs as part of the application binary; and (2)~its document-flexible
schema accommodates evolving conversation structures without schema migrations
as the application develops. SQLite would require more rigid schema design;
SurrealDB's flexible document model simplifies storing nested tool calls,
results, and metadata that may change across versions. Analytical datasets
are held in memory during sessions and not
persisted to the database, limiting storage to lightweight conversation
metadata.

\textbf{Handling unimplemented methods.} When users request capabilities
outside the implemented set, the LLM's behavior depends on its training.
In some cases, it maps the request to the closest available alternative
and explains the substitution. In others, it reports that the capability
is not available. The system does not explicitly enumerate unimplemented
methods; rather, the LLM infers scope from the tool schema. Users seeking
methods not in the tool list should verify availability before assuming
support.

\subsection{Identification diagnostics}

Beyond executing statistical methods, the system generates data-driven
warnings when estimation results suggest potential violations of identifying
assumptions. These warnings are non-blocking---results are always returned
to the user---but highlight conditions that warrant careful interpretation.

For \emph{instrumental variables}, the system checks first-stage
$F$-statistics against the Stock-Yogo threshold of
10~\citep{Stock+Yogo:2005}. Values below 10 trigger weak instrument warnings
with severity scaled by the degree of weakness ($F < 5$ generates critical
warnings). When models are overidentified, the Sargan test is automatically
computed and rejection ($p < 0.05$) triggers exclusion restriction warnings.

For \emph{difference-in-differences}, staggered designs include a pre-trend
test (joint test that pre-treatment ATT estimates are zero). For
\emph{matching and IPW}, balance diagnostics check whether post-matching
standardized mean differences exceed 0.1, the conventional threshold for
acceptable balance~\citep{Austin:2011}. For \emph{regression discontinuity},
effective sample sizes within the bandwidth are checked against minimum
thresholds ($n < 50$ per side triggers warnings).

Each warning includes: (1) a severity level (info, caution, warning, critical),
(2) a diagnostic message explaining the concern, (3) the specific assumption
at risk, and (4) suggested remediation steps. The \code{EconError} type
provides structured error information for estimation failures---singular
matrices, convergence problems, insufficient clusters---that enables the LLM
to provide actionable recovery guidance rather than generic failure messages.
