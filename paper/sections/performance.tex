\section{Performance benchmarks} \label{sec:performance}

This section presents detailed performance benchmarks comparing
\pkg{prompt2analytics} against established \proglang{R} packages across
approximately 75 method configurations spanning regression, econometrics,
machine learning, and forecasting. Benchmark methodology follows
Section~\ref{sec:comparison}; see Appendix~\ref{app:benchmark-methodology}
for complete details.%
\footnote{A notable omission is \pkg{fixest} \citep{Berge:2018}, which is
substantially faster than both \pkg{plm} and \pkg{lfe} for panel methods.
Informal benchmarks suggest \pkg{fixest} also outperforms \pkg{prompt2analytics}
for HDFE problems at large scale due to its highly optimized C++ implementation
of the alternating projections algorithm. Users prioritizing raw speed for
panel econometrics over natural language interfaces should consider \pkg{fixest}.}

\subsection{Benchmark results} \label{sec:benchmark-results}

We present benchmark results across five method categories: regression,
panel econometrics, discrete choice, machine learning, and data munging.
Sample sizes range from $n = 100$ to $n = 100{,}000$ observations.

Table~\ref{tab:benchmark-full} presents benchmark results at $n = 100{,}000$
observations, where computational costs dominate startup overhead.
\pkg{prompt2analytics} demonstrates substantial speedups for compute-intensive
methods: Panel Fixed Effects (6.1$\times$), OLS with robust standard errors
(5.1--5.5$\times$ for HC0--HC3), K-Means clustering (4.5$\times$), Panel
Random Effects (4.0$\times$), and Logit regression (3.1$\times$). Group-by
aggregation shows a 1.6$\times$ speedup. These speedups at $n = 100{,}000$
represent the most reliable performance comparison, as computational costs
dominate language overhead at this scale. Results at smaller sample sizes
(not shown) exhibit larger speedup factors that partly reflect \pkg{plm}'s
known inefficiencies at small $n$ rather than fundamental \proglang{Rust}
performance advantages.

Methods showing \proglang{R} outperforming \proglang{Rust} warrant detailed
explanation, as these apparent ``slowdowns'' illuminate the performance
characteristics of both approaches.

\textbf{Data manipulation operations} (filter, mutate, group-by aggregation)
show modest speedups or near-parity rather than the dramatic gains seen in
statistical methods. This reflects \proglang{R}'s mature, highly optimized
implementations for basic operations: \fct{dplyr} leverages vectorized C++
code that minimizes overhead for common transformations. The \pkg{polars}
library used in \pkg{prompt2analytics} provides competitive but not superior
performance for these operations. Users should not expect data wrangling to
be dramatically faster than established \proglang{R} workflows.

\textbf{PCA} showing a 0.3$\times$ speedup (i.e., \proglang{R} is faster)
reflects \proglang{R}'s use of highly optimized LAPACK routines for eigenvalue
decomposition. The \proglang{Rust} implementation via \pkg{faer} prioritizes
numerical stability and memory safety over raw speed for this operation.
For PCA as a dimensionality reduction step in larger workflows, this difference
is typically negligible; for workflows dominated by repeated PCA computations,
\proglang{R}'s \fct{prcomp} may be preferable.

\textbf{Probit regression} uses Newton-Raphson optimization in our implementation,
while \proglang{R}'s \fct{glm} benefits from decades of refinement in iteratively
reweighted least squares (IRLS). The difference represents implementation
maturity rather than fundamental language performance.

\textbf{OLS without robust standard errors} is sufficiently fast in both
languages that CLI startup overhead dominates computation time at moderate
sample sizes. The ``slowdown'' disappears when measured as pure computation
time (excluding process initialization). To quantify this: the \code{p2a}
CLI incurs approximately 12--15~ms of startup overhead (binary loading,
argument parsing, polars initialization), while \proglang{R}'s equivalent
overhead is approximately 50~ms but is excluded from \pkg{bench} timings
which measure only the timed expression. For methods completing in under
20~ms, the Rust CLI overhead dominates; for methods taking over 100~ms,
it becomes negligible. Table~\ref{tab:benchmark-full} reports wall-clock
time including CLI overhead; the MCP server interface avoids this overhead
entirely since the server runs persistently.

The general pattern is that \pkg{prompt2analytics} excels at computationally
intensive iterative methods (panel fixed effects, robust standard errors, HDFE)
where \proglang{Rust}'s compile-time optimizations and memory efficiency
compound across iterations. For single-pass operations dominated by optimized
linear algebra, \proglang{R}'s BLAS bindings provide comparable or superior
performance.

\subsubsection{Discussion}

Performance differences reflect \proglang{Rust}'s ownership model (reduced heap
allocation), SIMD vectorization via \pkg{faer}, compiled vs. interpreted
overhead, and column-based API (no formula parsing). Speedup factors decrease
with sample size as matrix operations dominate.
