\section{Performance and deployment} \label{sec:deployment}

The three principles require software fast enough to run locally and
models small enough to fit on consumer hardware. This section presents
performance benchmarks against \proglang{R} and describes fully local
deployment configurations where all components---including the language
model---run on the user's machine.

\subsection[Performance comparison with R]{Performance comparison with \proglang{R}}

Table~\ref{tab:benchmark-full} presents benchmark results across 41
method configurations at $n = 100{,}000$ observations with multiple
iterations per method---100 for fast methods, 50 for moderate, 20 for
slow---after 10 warmup iterations (median timing reported;
see Appendix~\ref{app:benchmark-methodology} for details). Both \proglang{R} and \proglang{Rust} timings measure
computation only, excluding data loading and process startup, ensuring
a fair comparison of algorithmic performance. Across 42 method
configurations, the median speedup is 4.1$\times$, with wide
variation by method category reflecting the relative weight of
interpreter overhead versus numerical computation.

The majority of methods show speedups of 2--10$\times$: OLS
(6.9$\times$), IV/2SLS (5.2$\times$), Hausman (4.7$\times$), MSTL
decomposition (4.4$\times$), Kaplan-Meier (4.1$\times$), and t-tests
(3--7$\times$). Compute-intensive iterative methods see larger gains:
quantile regression (44$\times$), Wilcoxon rank-sum (23$\times$),
OLS with robust standard errors (15--22$\times$ for HC0--HC3), panel
fixed effects (21$\times$), and DBSCAN (14.5$\times$). One extreme
outlier---Granger causality at 660$\times$---likely reflects
implementation differences in \proglang{R}'s
\code{lmtest::grangertest} (which fits full VAR models per lag) rather
than a genuine language advantage, and should be interpreted with
caution.

Methods near parity include two-way ANOVA, AR, and negative binomial
regression (all ${\sim}1\times$). \proglang{R} outperforms
\proglang{Rust} for PCA (0.03$\times$), NLS (0.1$\times$),
diagnostics (0.2$\times$), and ACF (0.4$\times$)---operations where
\proglang{R}'s highly optimized LAPACK bindings or vectorized C
implementations dominate. For high-dimensional fixed effects,
\pkg{fixest}~\citep{Berge:2018} provides a highly optimized C++
implementation that likely outperforms both \pkg{plm} and
\pkg{prompt2analytics} at large scale; our HDFE benchmark against
\pkg{plm} shows only 1.3$\times$ speedup. At smaller sample sizes
($n < 1{,}000$), speedups can be much larger---for example, panel
fixed effects at $n = 100$ is $184\times$ faster---but these figures
primarily reflect \proglang{R}'s interpreter overhead (formula
parsing, factor handling) rather than computational advantages.
Complete benchmark results are in
Appendix~\ref{app:benchmark-methodology}; reproduction instructions
are in Appendix~\ref{app:reproducibility}.

The performance advantages stem from \proglang{Rust}'s ownership system
(stack allocation and in-place mutation without garbage collection),
the \pkg{faer} library's SIMD-optimized linear algebra, the
column-based API that eliminates formula parsing, and the
single-language design that avoids data serialization between
components. \pkg{prompt2analytics} performs best on computationally intensive
iterative methods (panel fixed effects, robust standard errors,
quantile regression, DBSCAN) where compile-time optimizations and
memory efficiency compound across iterations. For single-pass
operations dominated by optimized linear algebra (PCA, ACF, NLS),
\proglang{R}'s BLAS bindings provide comparable or superior
performance.

For users with CUDA-capable hardware, \pkg{p2a-core} provides optional
GPU acceleration via cuBLAS and cuSOLVER. The \code{cuda} feature flag
enables GPU code paths at compile time; at runtime, a singleton context
attempts CUDA initialization and falls back silently to CPU if no
device is available. Public function signatures are unchanged---users
do not modify their code.

Not all operations benefit from GPU offloading. Transfer overhead
between host and device memory dominates for small problems, and
bandwidth-bound operations (e.g., matrix-vector products) can be
faster on CPU even for large $n$. The dispatch logic therefore checks
two conditions before routing to GPU: a minimum problem dimension
(e.g., $k \geq 30$ columns for $\bm{X}'\bm{X}$) and a minimum
computational intensity (e.g., $nk^2 \geq 2 \times 10^7$). Both
thresholds are configurable via environment variables, allowing
users to tune dispatch for their specific hardware. A shape guard
additionally prevents GPU dispatch for tall-skinny matrix
multiplications, where the CPU path is substantially faster.

Table~\ref{tab:gpu-ops} summarizes the operations with GPU code paths
and observed speedups on an NVIDIA DGX~Spark (Grace Blackwell, 12~GB
VRAM, unified memory). The largest gains appear in $\bm{X}'\bm{X}$
computation (up to $7\times$ for $k \geq 100$) and PCA via the
covariance path (up to $13\times$ for tall matrices). K-means distance
computation and sandwich estimators for heteroskedasticity-robust
standard errors see more modest improvements of $2$--$3\times$.

Most econometric applications involve moderate numbers of regressors
($k < 30$), where CPU execution is faster. GPU acceleration is
therefore not part of the default build; it targets high-performance
computing scenarios with many covariates, high-dimensional clustering,
or large-scale bootstrap procedures. The implementation exploits a
layout convention---\pkg{ndarray}'s row-major storage appears as a
transposed column-major matrix to cuBLAS---that avoids data
reordering on the host side. Operations that are bandwidth-bound
rather than compute-bound ($\bm{X}'\bm{y}$ via DGEMV) remain
on CPU, where OpenBLAS is consistently faster. The feature is
currently limited to NVIDIA GPUs via CUDA; extension to other
backends (e.g., Vulkan compute, Metal) is left for future work.

\subsection{Fully local deployment} \label{sec:local}

While cloud-hosted language models offer convenience, many use cases
require complete data isolation. The fully local configuration runs all
components on the user's machine via Ollama~\citep{Ollama}, eliminating
all external network communication. Appendix~\ref{app:deployment}
provides platform-specific installation instructions.

The evaluation results (Section~\ref{sec:evaluation}) inform practical
model recommendations. Open-source models can match proprietary
alternatives: Qwen 2.5 72B achieves 100\% accuracy on our benchmark,
identical to Claude 3.5 Haiku. Users requiring complete data isolation
need not sacrifice tool selection quality.

Ministral 3B demonstrates that even very small models achieve viable
accuracy. With only 3 billion parameters, it runs on virtually any
modern laptop---approximately 6~GB VRAM for GPU inference, or CPU-only
with 8~GB RAM. The 9.2\% error rate concentrates in genuinely ambiguous
cases. The 7--8B parameter class offers a practical middle ground,
fitting on consumer GPUs with 8+ GB VRAM (e.g., NVIDIA RTX 3070/4070)
or Apple M-series devices with 16~GB unified memory.

Beyond LLM memory, users must budget RAM for dataset operations.
A typical configuration analyzing 100,000 observations with a 7B
model requires approximately 8--10~GB total RAM; a minimal laptop
setup (Ministral 3B, small datasets) works within 16~GB.
Appendix~\ref{app:hardware-requirements} provides detailed RAM
requirements by dataset size
(Table~\ref{tab:hardware-data}) and system configurations for
common deployment scenarios (Table~\ref{tab:hardware-summary}).

\subsection{Privacy guarantees}

The fully local configuration provides complete data isolation. No
network requests are made to external services. All model inference
occurs on the user's hardware, dataset contents never leave the
machine, and query history remains in local storage. This architecture
is suitable for analysis of sensitive data where regulatory requirements
(GDPR, HIPAA, or institutional review board protocols) or organizational
policy prohibit cloud processing. Users should verify that their Ollama
installation does not enable telemetry or automatic model updates.
Appendix~\ref{app:privacy} provides detailed data transmission
documentation and security best practices. The next section discusses limitations and future directions.
